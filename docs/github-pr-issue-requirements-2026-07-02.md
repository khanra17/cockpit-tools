# GitHub PR / Issue 需求整理

整理时间：2026-07-02

数据范围：

- 开放 PR：31 个
- 开放 issue：339 个
- 已抓取有评论 issue 的评论：255 条

说明：

- 当前版本已重构，开放 PR 的代码不建议直接合并；以下只把 PR/issue 还原成需求、缺陷和验收方向。
- `P0` 表示阻断/数据损坏/大面积不可用，`P1` 表示高频主路径或强需求，`P2` 表示平台扩展或体验增强，`P3` 表示低风险优化、文档或支持项。
- 开发前仍需按当前代码和真实客户端行为重新调查，尤其是账号切换、平台包、provider gateway、会话恢复等链路。

## 优先级总览

建议先处理：

1. 平台包/adapter 启动失败、新版本无法识别已安装平台、按钮遮挡等 v1.0.0 级回归。
2. Codex / Antigravity / API 服务切号与配置持久化问题，避免“界面成功但实际未切换”。
3. Codex provider 配置、模型下拉、URL/API key 名称丢失，避免用户配置被覆盖。
4. 会话恢复/可见性/跨设备迁移，避免用户历史记录丢失或必须手工修复。
5. 大账号量卡死、导入/删除/筛选批处理，避免 500+ 账号场景不可用。

## P0 阻断与数据正确性

### REQ-001 平台包与 adapter 运行健康

来源：#1363, #1364, #1348, #1299, #1296, #1366, PR #1320, PR #1297

需求：新平台包架构下，平台页必须能正确识别已安装平台包、启动对应 adapter、读取启动信息，并在失败时给出可操作错误。平台操作区的“检查更新/卸载”等按钮不能展开后遮挡其他操作且无法收回。

验收要点：

- 已安装平台包在侧边栏和平台页不误判为“未安装”。
- adapter 启动信息为空、EOF、RPC 端口不可达、schema 缺字段时，弹框/操作区显示明确错误和重试入口。
- Windows sidecar/cliproxy 缺失时 updater fallback 选择 MSI metadata，不误选 NSIS。
- 平台包操作区在窄宽度下不遮挡主操作按钮，可收起或自适应换行。

### REQ-002 新版本性能、大账号量和低动画模式

来源：#1362, #1152, #1114, #1106, #1124, #1128, #1140, #829, #723, PR #1294, PR #686

需求：降低主界面、账号列表、大批量账号场景和托盘后台状态的内存/CPU/渲染压力。

验收要点：

- 500/1000+ 账号加载、筛选、刷新、删除不会卡死主线程。
- 关闭到托盘时可释放 WebView 或显著降低驻留内存，重新打开能恢复页面导航目标。
- 提供全局“减少动画”设置，关闭装饰性动画、毛玻璃、阴影和高频重算。
- 批量删除、批量刷新、导入检查走后台任务，不阻塞 API 服务或 UI。

### REQ-003 Codex 账号切换、token 刷新与官方 auth 兼容

来源：#891, #952, #1038, #1020, #1126, #1167, #1178, #1182, #1328, #1335, #666, #836, #877, #919, #907, #1050, #1058, #827, #828, PR #1325, PR #994, PR #980

需求：Codex 切号必须按官方客户端真实落盘规则写入 `auth.json` / `config.toml` / profile 状态，处理 refresh token 轮换、`auth_mode=chatgpt`、access-token-only 绑定、Team/Plus 多账号共存等场景。

验收要点：

- 导入当前选中账号的新 token 后，运行态立即重新激活该账号。
- ChatGPT/OAuth 账号写入官方可识别的 `auth_mode` 和 token 结构。
- token 被 CPA/官方客户端刷新后，Cockpit 能同步新 token，避免 `refresh_token_reused`。
- 切号成功判定必须读取实际落盘与运行态，不能只改前端状态。
- Windows/macOS 默认 Codex 实例按官方默认上下文识别，不强制 `CODEX_HOME`。

### REQ-004 Codex API Key / provider 配置持久化不被覆盖

来源：#1365, #1278, #1346, #1216, #1194, #1060, #1044, #1053, #1029, #1271, #1043, #1289, #1285, #1292, #1332, PR #1334

需求：保存、切换、启动 API 服务或绑定 OAuth 时，不得覆盖用户自定义 API key 名称、Base URL、`model_provider`、插件配置或 `model_catalog_json`。显式 `false` 必须持久化，不能被迁移逻辑当成旧数据重置。

验收要点：

- 同一供应商多个 key 的展示名保存后不被供应商名覆盖。
- Base URL 不因重启、切号、API 服务启动被清空或回滚官方地址。
- 绑定 OAuth 的 `bound_oauth_use_local_gateway=false` 能序列化并重读保持 false。
- 切回非 API_KEY provider 时只清理 Cockpit 自己管理的 catalog 指针，保留用户自定义配置。

### REQ-005 Antigravity / Antigravity IDE 真实路径与切号

来源：#1361, #1291, #976, #966, #911, #1074, #1077, #1001, #1142, #818, #1024, PR #1314, PR #1243

需求：Antigravity 与 Antigravity IDE 在 Windows/macOS 上要区分数据目录、快捷方式、进程和真实当前账号，避免 UI 显示已切换但 IDE 实际仍是浏览器本地账号。

验收要点：

- `%APPDATA%\Antigravity` 与 `%APPDATA%\Antigravity IDE` 同时存在时，优先选择包含 `state.vscdb` 的正确目录。
- 启动快捷方式不因名称包含关系误开 IDE/非 IDE 另一个程序。
- 当前账号显示读取运行时凭据，不 fallback 到“最后一次切换记录”。
- 切号失败时说明缺失的是数据库、凭据、进程还是路径配置。

### REQ-006 API 服务协议、路由与错误兼容

来源：#1324, #1176, #1301, #1321, #1230, #981, #896, #1172, #1204, #900, #870, #1109, #1125, #912, #967, #795, #1047, #1051, #961, #728, #1215, #856, #1070, PR #932

需求：API 服务需要更完整兼容 OpenAI Responses、Chat Completions、Anthropic/Claude Code、WSL/LAN 访问、上下文压缩、图片生成尺寸等路径，并清晰区分上游 401/403/429/502/504 与本地网关错误。

验收要点：

- 支持误把 `/v1/responses` 拼到 `/v1/chat/completions` 后面的兼容路径。
- 请求体过大、stream 中断、compact context、上游超时要有明确错误分类。
- GPT image 尺寸和上游能力限制要按真实响应处理，不能虚报 2K/4K 能力。
- 可选支持 Anthropic-compatible 路由，并明确模型/工具 schema 转换边界。
